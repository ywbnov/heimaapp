# 黑马记账

## 1. 项目定位

黑马记账是一款运行在 Windows 和 macOS 电脑上的个人记账应用。它的第一目标是让用户在几秒内记录一笔人民币花销，并能按日期、一级分类、二级分类查看和分析消费情况。

本文件是项目的首版产品文档和协作约定。后续开发以这里确认过的内容为基线；每次做出重要决定后，都要在本文档的“决策记录”中补充。

## 2. 重要协作约定（必须遵守）

用户不具备编程背景，因此在整个项目开发期间：

1. 任何技术问题（技术栈、数据库、打包方式、数据同步、权限、安全、发布方式等）都必须先列出 2-3 个可行方案。
2. 每个方案必须用通俗语言解释：它是什么、对用户有什么影响、优点、缺点、成本或风险。
3. 必须明确给出推荐方案及推荐理由，但最终选择权由用户决定。
4. 未经用户明确确认，不得擅自做出会影响项目方向、数据格式、跨平台能力或后续维护成本的技术决定。
5. 对不影响方向的小决定可以先采用稳妥默认值，但要说明这是默认值，并记录在“决策记录”中。
6. 技术说明应先讲结论和用户可感知的影响，必要时再补充技术细节，避免只使用专业术语。
7. Windows 和 macOS 是硬性支持范围。任何方案如果不能同时支持两者，必须明确标记为不满足要求。
8. 每个开发阶段开始前，先说明目标、方案和验收方式；阶段完成后说明实际结果、已知限制和下一步需要用户决定的事项。

## 3. 目标用户与目标

### 目标用户

- 想在电脑上管理日常支出的个人用户。
- 希望不依赖网络也能记账，并能随时导出自己数据的用户。
- 不需要复杂财务专业知识，希望界面直观、输入快速的用户。

### 产品目标（首个版本）

- 新用户无需学习即可完成第一笔记账。
- 常用记账操作在 10 秒内完成。
- 支持人民币金额和两级消费分类。
- 支持按时间和分类查询，并提供基础汇总统计。
- 数据默认保存在本机，断网时仍可使用。
- 能够备份、恢复和导出数据，避免用户被应用锁定。

### 首版不包含的内容

- 银行或支付平台自动拉取账单。
- 多人共享账本和在线账户体系。
- 投资、借贷、发票等专业财务管理。
- 复杂预算算法或自动化理财建议。

## 4. 核心功能

### 4.1 记录一笔花销

必填字段：

- 金额：人民币，精确到分，必须大于 0。
- 日期和时间：默认当前时间，可修改。
- 分类：选择一个一级大类和一个二级小类。

可选字段：

- 备注：例如“周末聚餐”。
- 支付方式：现金、银行卡、微信、支付宝、其他。

操作要求：

- 保存成功后给出明确反馈，并保留“继续记一笔”的快捷入口。
- 金额输入应优先使用数字键盘，自动显示人民币格式。
- 未保存退出时提示用户，避免误丢数据。

### 4.2 两级分类

首版提供以下默认分类。用户可以新增、重命名、停用分类；已经使用过的分类不能被直接删除，只能停用，以保证历史账目仍可正确显示。

| 一级大类 | 二级小类 |
| --- | --- |
| 餐饮 | 早餐、午餐、晚餐、外卖、零食饮料、咖啡茶饮 |
| 交通 | 公交地铁、打车网约车、加油充电、停车、高铁飞机 |
| 住房 | 房租、物业、水费、电费、燃气、网络通信、家居维修 |
| 日用购物 | 日用品、服饰鞋包、数码电器、家居用品、宠物用品 |
| 娱乐休闲 | 电影演出、游戏、运动健身、旅游度假、兴趣爱好 |
| 医疗健康 | 挂号就医、药品、保健、牙科眼科 |
| 教育成长 | 课程培训、书籍资料、考试证书 |
| 社交人情 | 聚餐请客、礼物红包、婚礼寿宴 |
| 金融服务 | 手续费、保险、会员订阅、利息税费 |
| 其他 | 未分类、临时支出 |

说明：分类名称和数量可在后续用户测试中调整；分类调整不应改变历史记录的金额和日期。

### 4.3 账目列表

- 默认按时间倒序展示。
- 支持按日期范围、一级分类、二级分类、支付方式筛选。
- 支持编辑和删除；删除前二次确认。
- 每日显示小计，并在列表顶部显示筛选结果总额和笔数。
- 空状态要告诉用户如何开始，而不是只显示空白页面。

### 4.4 统计概览

- 今日、本周、本月、本年度支出总额。
- 按一级分类汇总金额和占比。
- 点击一级分类可展开二级分类明细。
- 提供最近 6 个月的月度支出趋势。
- 统计结果必须与账目列表使用同一套筛选条件，避免数字不一致。

### 4.5 数据管理

- 本地自动保存。
- 手动导出 CSV（方便用 Excel 打开）和 JSON（完整备份）。
- 从 JSON 备份恢复，恢复前显示将要覆盖或合并的数据范围。
- 提供“备份位置”和“最近备份时间”提示。
- 首版默认单机使用；云同步属于后续可选功能，不能阻塞首版发布。

## 5. 页面与用户流程

### 页面

1. **首页/概览**：本月支出、快捷记账按钮、最近账目、分类概览。
2. **记一笔**：金额、分类、日期时间、备注、支付方式，支持快速保存。
3. **账目**：可搜索、筛选、编辑、删除历史记录。
4. **统计**：时间趋势和分类占比，可切换统计周期。
5. **设置**：分类管理、数据导入导出、备份、主题和基础偏好。

### 最重要的流程

打开应用 -> 点击“记一笔” -> 输入金额 -> 选择一级/二级分类 -> 可选填写备注 -> 保存 -> 回到首页并看到最新记录。

## 6. 数据设计（产品层面）

每一笔支出至少包含以下信息：

- 唯一编号。
- 金额（人民币分值，避免小数计算误差）。
- 发生时间。
- 一级分类编号和名称快照。
- 二级分类编号和名称快照。
- 备注。
- 支付方式。
- 创建时间、最后修改时间。
- 是否已删除（用于保护历史统计和备份一致性）。

分类也需要独立保存：编号、父级编号、名称、是否启用、排序顺序、创建时间和更新时间。

## 7. Windows 与 macOS 要求

- 提供 Windows 安装包和 macOS 安装包。
- 首版支持主流 64 位 Windows 和近年主流 macOS；具体最低版本在技术方案确定后再确认。
- 界面使用鼠标、键盘均可完成主要操作。
- 不依赖持续联网；联网不是记账必需条件。
- 安装、升级、卸载过程不应删除用户账目，除非用户明确选择清除数据。
- 需要在两种系统上分别验证金额、日期、导入导出、备份恢复和窗口缩放。

## 8. 技术栈候选方案与已确认方案

以下方案都可以实现 Windows + macOS 桌面应用。它们的差异主要体现在安装包大小、运行效率、开发维护方式和未来扩展成本。

### 方案 A：Electron + React + TypeScript + SQLite（推荐起点）

**通俗解释**：用成熟的网页界面技术做桌面应用，Electron 负责把网页界面装进 Windows/macOS 应用，SQLite 负责本地保存数据。

优点：

- 资料和开源组件最多，适合快速做出可用版本。
- 表格、筛选、图表等记账界面开发效率高。
- Windows/macOS 打包流程成熟，后续找人维护相对容易。

缺点：

- 安装包和内存占用通常比原生应用大。
- 需要同时理解桌面进程和界面代码，安全配置要认真处理。

适合：优先快速验证产品、功能会逐步增加的项目。

### 方案 B：Tauri + React/Vue + SQLite

**通俗解释**：界面仍使用网页技术，但桌面外壳更轻量，尽量使用系统自带能力。

优点：

- 安装包更小、内存占用通常更低。
- 桌面安全边界清晰，适合长期维护。
- 仍能复用大量前端界面组件。

缺点：

- 需要引入 Rust 这一门额外技术，开发和排查问题的门槛更高。
- 某些系统级能力和第三方组件的资料少于 Electron。

适合：重视轻量、性能和长期桌面体验，能够接受更高初期技术成本的项目。

### 方案 C：Flutter Desktop + Dart + SQLite

**通俗解释**：用 Flutter 一套界面代码同时绘制 Windows 和 macOS 的桌面界面。

优点：

- 跨平台界面一致，动画和交互体验好。
- 代码结构统一，适合未来同时扩展到手机端。
- 安装包通常比 Electron 小。

缺点：

- 需要使用 Dart/Flutter，网页生态中的现成组件不能直接复用。
- 桌面端细节（系统菜单、窗口行为、无障碍等）需要更多专项验证。

适合：明确希望未来做 iOS/Android，并愿意围绕 Flutter 建立完整技术体系的项目。

### 初步建议

如果首要目标是尽快得到一个稳定可用的电脑记账软件，我建议先选方案 A；如果你更看重安装包体积和长期运行效率，可选方案 B；如果未来一定要同步开发手机端，可重点考虑方案 C。

### 已确认的方案 A 技术边界

- **Electron 主进程**：负责创建窗口、应用生命周期、系统菜单、文件导入导出和安全地调用本地能力。
- **React + TypeScript 渲染界面**：负责首页、记账表单、账目列表、统计和设置等用户看到的内容。
- **SQLite 本地数据库**：负责保存账目、分类和应用设置；首版不依赖云端服务。
- **跨平台打包**：同一套代码分别生成 Windows 安装包和 macOS 安装包，发布和最低系统版本将在实现前再次说明。
- **构建与打包工具**：采用 `electron-vite + electron-builder`，前者负责开发和构建，后者负责生成 Windows/macOS 安装包。
- **SQLite 驱动**：采用 `better-sqlite3`，用于本地账目、分类和设置的稳定读写；打包阶段需要验证 Windows/macOS 原生模块兼容性。
- **界面组件**：采用 `Ant Design`，用于表单、表格、日期选择、弹窗和基础反馈；在其基础上定制黑马记账的颜色和布局。
- **自动化测试**：采用 `Vitest`，覆盖金额校验、分类规则、统计计算、本地数据访问和关键界面行为。
- **统计图表**：采用 `Recharts`，用于分类占比、月度趋势等首版统计图表；只实现记账所需的基础图表。
- **原生模块兼容**：`better-sqlite3` 在测试时按当前 Node.js ABI 重建，在 Electron 构建前按 Electron ABI 重建；npm 脚本会自动完成这两次切换，避免跨运行时加载错误。
- **Windows 渲染兼容**：窗口保持 `contextIsolation` 和 `nodeIntegration` 安全配置，同时关闭窗口级 Chromium 沙箱，以兼容部分受限桌面、远程桌面或虚拟机环境中的渲染进程启动限制；应用只加载本地打包资源，不加载远程网页。
- **当前 MVP 实现状态**：已完成本地记账、两级分类、首页概览、账目列表（关键词/日期/分类/支付方式筛选）、删除、基础统计、CSV 导出、JSON 备份恢复和设置页分类停用。分类新增/重命名、账目编辑和 macOS 实机打包验证列入下一轮迭代。
- **数据边界**：渲染界面不能直接访问文件系统或数据库，必须通过 Electron 提供的受控接口访问，以降低数据损坏和安全风险。

## 9. 首版验收标准

- 在 Windows 和 macOS 上均能安装、启动和卸载。
- 可以成功新增、编辑、删除一笔花销，金额单位为人民币且精确到分。
- 每笔记录必须关联一个一级分类和一个二级分类。
- 账目筛选结果、首页汇总和统计页面的数字一致。
- 关闭并重新打开应用后，已保存数据仍然存在。
- CSV 导出文件可被常见表格软件打开；JSON 可成功恢复。
- 停用分类后，旧账目仍可查看，新账目不能误选该分类。
- 主要操作不依赖网络，窗口缩放后界面不重叠、不截断。

## 10. 分阶段计划（技术方案确认后执行）

1. **产品确认**：确认技术方案、首版分类、是否需要支付方式字段和默认备份策略。
2. **交互原型**：确定首页、记一笔、账目、统计、设置五个页面的布局和操作流程。
3. **MVP 开发**：完成本地记账、两级分类、账目列表、基础统计、导入导出。
4. **跨平台验证**：分别在 Windows 和 macOS 验证安装、升级、数据保存和窗口行为。
5. **试用与调整**：根据真实使用反馈优化分类、快捷操作和统计展示。

## 13. 当前可用版本

### Windows 安装

Windows 安装程序已生成：`release/黑马记账 Setup 0.1.0.exe`。

当前安装包是 Windows x64 NSIS 安装程序，适用于常见的 64 位 Windows 10/11 环境；不支持 32 位 Windows，其他旧版本 Windows 尚未承诺兼容。直接双击安装程序即可安装。账目保存在当前 Windows 用户的应用数据目录中，正常关闭、升级或卸载应用时不会主动删除数据库文件。

此前生成的安装包曾因 `better-sqlite3` 与 Electron 的原生 ABI 不匹配而出现“安装后点击打开无反应”。请使用本次重新生成的安装包，不要继续使用旧文件。重新生成前已执行 Electron 原生模块重建，并在 Windows unpacked 版本中验证了主进程、数据库初始化和窗口创建。

### macOS 状态

项目已配置 macOS DMG 打包目标，但当前开发环境是 Windows，不能在这里完成 macOS 实机打包、代码签名和运行验证。发布 Mac 版本前必须在 macOS 设备或 macOS 构建环境中执行：

1. 安装项目依赖。
2. 运行完整自动化测试和 TypeScript 类型检查。
3. 生成 DMG 安装包。
4. 验证新增、筛选、删除、统计、备份恢复以及关闭重启后的数据持久化。

### 当前验证结果

- Vitest：9 个测试文件、18 个测试通过。
- TypeScript：主进程和界面/测试两套类型检查通过。
- Electron-Vite：生产构建通过。
- electron-builder：Windows x64 NSIS 安装包生成通过。
- Windows unpacked 运行验证：启动日志显示数据库初始化和窗口创建完成，无 `better-sqlite3` ABI 错误。
- macOS：仅完成构建配置，尚未实机验证。
